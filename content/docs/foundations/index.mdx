---
title: Foundations
description: The design principles and responsibility boundaries shared across Zagvar projects.
---

Trading products vary widely. A retail broker, exchange, institutional terminal,
and self-custodial crypto interface should not be forced into one complete
product model.

Zagvar provides smaller pieces that can be adopted independently:

```txt
market-data providers
        ↓
      Relay
        ↓
application state and policy
        ↓
      Mosaic
        ↓
trading interface

@zagvar/decimal preserves exact economic values across these layers.
```

The diagram shows a common composition, not a required deployment topology.
Mosaic can consume data from any backend, Relay can serve any client, and
Decimal can be used without either project.

## Exact decimal values

Prices, quantities, notionals, balances, fees, and volumes must survive JSON,
caching, state updates, and calculations without silent floating-point changes.
Decimal defines the canonical string format and exact arithmetic used by Mosaic
and Relay.

Keep those values as strings through business and transport boundaries. Format
them for people only at the presentation edge.

## Provider-neutral boundaries

External providers use different symbols, field names, units, timestamps, and
transport protocols. Provider-specific details should be translated at the edge
of the application rather than propagated throughout the product.

Relay normalizes upstream market data into readable contracts. Mosaic accepts
presentation-ready values and order rules without needing to know which
provider, exchange, or backend produced them.

## Shared market identity

Mosaic and Relay use compatible semantics for the fields that naturally cross
the frontend/backend boundary:

| Field | Meaning |
| --- | --- |
| `symbol` | Application- or provider-normalized instrument identifier |
| `assetClass` | Economic category such as equity, crypto, FX, or fund |
| `venue` | Exchange, venue, or source market when relevant |
| `baseAsset` | Base side of a pair-based instrument |
| `quoteAsset` | Asset in which the pair is priced |

Prices use full names such as `price`, `bidPrice`, and `askPrice`. Quantities are
expressed with `quantity` rather than provider-shaped abbreviations. Public
timestamps are ISO 8601 strings.

Compatible meaning does not remove the application mapping step. An application
may still enrich, filter, aggregate, authorize, or relabel data before
presenting it.

## Venue-aware data

The same symbol can have different prices and liquidity on different venues.
Relay therefore treats optional `venue` as part of market-data identity for
quotes, trades, bars, and order books.

Omitting venue represents a deliberate default or consolidated stream. It does
not mean “every venue.” Applications decide how consolidated data is created and
how it should be labelled in the interface.

## Backend authority

Mosaic validation improves responsiveness and helps users form valid order
intents, but frontend validation is never a security or execution boundary.

Before placing an order, the backend remains responsible for validating current
market state, account state, balances, buying power, permissions, fees, product
rules, and applicable compliance requirements.

## Deliberate product boundaries

Zagvar does not attempt to provide a complete trading platform. The application
continues to own:

- authentication and identity
- accounts, portfolios, and balances
- order routing, execution, and lifecycle persistence
- risk, permissions, and compliance
- entitlements and market-data licensing
- deposits, withdrawals, custody, and settlement
- product-specific workflows and visual identity

These responsibilities depend heavily on business model, jurisdiction,
providers, and operational requirements. Keeping them outside Mosaic and Relay
makes both foundations useful across more products without minimizing their
importance.
